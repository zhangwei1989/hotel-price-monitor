## 3. 技术方案对比

### 3.1 积分房监控技术方案

由于**没有官方API支持积分房查询**,只能采用自动化模拟方案:

#### 方案A: 浏览器自动化 (Puppeteer/Playwright)

**原理**: 模拟用户登录酒店官网,自动查询积分房

**优势**:
- ✅ 可以获取积分房真实数据
- ✅ 灵活性高,可适配任何官网
- ✅ 开源工具,成本低

**劣势**:
- ⚠️ 需要用户提供会员账号密码
- ⚠️ 官网改版需更新爬虫脚本
- ⚠️ 可能被反爬虫机制阻止
- ⚠️ 查询速度较慢(每个酒店5-15秒)
- ⚠️ 需要维护登录状态(Cookie管理)

**技术实现**:
```javascript
// 示例: 万豪积分房查询
async function queryMarriottPoints(credentials, hotelId, checkIn) {
  const browser = await playwright.chromium.launch({ headless: true });
  const page = await browser.newPage();
  
  // 1. 登录万豪账号
  await page.goto('https://www.marriott.com/login');
  await page.fill('#username', credentials.email);
  await page.fill('#password', credentials.password);
  await page.click('#signin-button');
  
  // 2. 搜索酒店
  await page.goto(`https://www.marriott.com/hotels/hotel-information/${hotelId}`);
  await page.fill('#check-in', checkIn);
  
  // 3. 切换到积分房查询
  await page.click('[data-testid="points-toggle"]');
  
  // 4. 提取积分房价格
  const pointsPrice = await page.textContent('.points-price');
  
  return { points: parseInt(pointsPrice), available: true };
}
```

**风险**:
- 🚨 违反部分网站的服务条款(需审慎评估法律风险)
- 🚨 账号可能被封禁

#### 方案B: 手动查询 + 人工录入

**原理**: 用户手动查询积分房,将价格录入飞书表格,系统仅做监控提醒

**优势**:
- ✅ 完全合规,无法律风险
- ✅ 无需维护爬虫脚本
- ✅ 数据准确性由用户保证

**劣势**:
- ❌ 需要人工操作,无法完全自动化
- ❌ 实时性差

**适用场景**: 监控任务数量少(10个以内)

#### 方案C: API代理服务(第三方付费)

**原理**: 使用专业的酒店数据服务商API

**可能的服务商**:
- 飞常准 (VariFlight) - 有酒店数据服务
- 数据堂 - 旅游数据API
- 快速数据(已关闭酒店积分房服务)

**优势**:
- ✅ 稳定性高
- ✅ 无需自己维护爬虫

**劣势**:
- ⚠️ **目前市场上很少有提供积分房数据的服务**
- ⚠️ 成本高(可能$1-5/次查询)
- ⚠️ 数据覆盖不全

**调研结果**: 🔴 **未找到成熟的积分房API服务商**

## 4. 推荐技术方案

### 4.1 混合方案(最优)

#### **阶段一: MVP (1个月上线)**

**现金房监控**:
- ✅ 使用 **Amadeus API** 查询酒店集团现金房
- ✅ 使用 **RapidAPI (Booking.com)** 补充数据
- ✅ 对于未覆盖的平台(香格里拉/GHA),使用 **Playwright爬虫**

**积分房监控**:
- ✅ 使用 **Playwright浏览器自动化**
- ✅ 用户提供会员账号(加密存储)
- ✅ 每日定时查询(避免频繁触发反爬)
- ✅ 失败自动重试机制

**飞书集成**:
- ✅ 飞书多维表格存储监控任务
- ✅ 飞书消息卡片推送告警
- ✅ 飞书自动化触发查询(可选)

#### **阶段二: 优化 (2-3个月)**

- 申请携程/飞猪官方API(如能通过)
- 优化爬虫稳定性(IP代理池、验证码识别)
- 添加价格趋势图表
- 支持多用户

---

### 4.2 技术栈选择

| 组件 | 技术 | 理由 |
|------|------|------|
| 后端服务 | **Node.js + Express** | 异步处理好,生态丰富 |
| 浏览器自动化 | **Playwright** | 比Puppeteer更稳定,支持多浏览器 |
| 定时任务 | **node-cron** | 轻量级,易于配置 |
| 数据库 | **PostgreSQL** (或MongoDB) | 存储监控任务、价格历史 |
| 缓存 | **Redis** | 登录状态管理、防重复查询 |
| 飞书SDK | **@larksuiteoapi/node-sdk** | 官方SDK |
| 第三方API | **Amadeus API** | 现金房主力数据源 |

## 9. 风险评估与应对

### 9.1 技术风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|---------|
| 积分房爬虫被封 | 高 | 中 | IP代理池 + 降低查询频率 + 多账号轮换 |
| 官网改版导致爬虫失效 | 中 | 高 | 监控告警 + 快速修复机制 |
| Amadeus API额度不够 | 中 | 低 | 升级企业版 或 降低查询频率 |
| 飞书API限流 | 低 | 低 | 批量推送 + 控制频率 |

### 9.2 法律风险

| 风险 | 评估 | 应对 |
|------|------|------|
| 爬虫违反服务条款 | ⚠️ 中等风险 | 仅供个人使用,不公开,不商业化 |
| 用户账号被封 | ⚠️ 中等风险 | 提前告知用户风险,签署免责协议 |

**建议**: 
- ✅ 仅供个人/内部使用,不对外销售
- ✅ 控制查询频率,模拟正常用户行为
- ✅ 用户自愿提供账号,明确风险

### 9.3 业务风险

| 风险 | 应对 |
|------|------|
| 监控任务过多导致查询缓慢 | 优先级队列 + 异步处理 |
| 价格数据不准确 | 多数据源交叉验证 |

## 10. 结论与建议

### 10.1 核心结论

#### ✅ 技术可行性: **可行**

- **现金房监控**: 通过Amadeus + RapidAPI可以实现,成熟度高
- **积分房监控**: 只能通过浏览器自动化实现,有一定风险但可控

#### ⚠️ 积分房监控挑战

- 🔴 **没有官方API支持**
- 🔴 **必须使用爬虫方案**
- 🔴 **存在被封风险**
- 🟡 **需要用户提供会员账号**

#### ✅ 推荐方案

**混合方案**:
- 现金房: Amadeus API (主) + RapidAPI (辅) + Playwright (补充)
- 积分房: Playwright浏览器自动化
- 飞书: 多维表格 + 消息推送

---

### 10.2 建议的实施路径

#### 路径A: 激进方案(完整功能,高风险)

**特点**: 积分房 + 现金房全覆盖
**风险**: 爬虫可能被封
**适合**: 愿意承担风险,追求完整功能

**实施**:
1. Week 1-4: 开发MVP(含积分房爬虫)
2. 小规模测试2周
3. 根据测试结果决定是否全面上线

#### 路径B: 保守方案(仅现金房,低风险) ⭐推荐

**特点**: 先上线现金房监控,积分房手动录入
**风险**: 低
**适合**: 追求稳定性,逐步迭代

**实施**:
1. Week 1-3: 开发现金房监控(Amadeus + RapidAPI)
2. 用户手动查询积分房,录入飞书表格
3. 系统监控价格变化,触发告警
4. 后期逐步测试积分房爬虫

#### 路径C: 混合方案(推荐) ⭐⭐⭐

**特点**: 现金房API + 积分房爬虫,但控制风险
**风险**: 中
**适合**: 平衡功能与风险

**实施**:
1. 现金房使用API(稳定)
2. 积分房使用爬虫,但:
   - 仅监控重要任务(≤10个)
   - 查询频率降低(每日1-2次)
   - 使用代理IP
   - 多账号轮换

---

### 10.3 最终建议

#### 🎯 我的推荐: **路径C (混合方案)**

**理由**:
1. ✅ 现金房通过API实现,稳定可靠
2. ✅ 积分房通过爬虫实现,但控制规模和频率
3. ✅ MVP 1个月上线,快速验证需求
4. ✅ 成本可控($10-60/月 + 服务器¥100-250/月)
5. ✅ 风险可控(小规模使用,不公开)
